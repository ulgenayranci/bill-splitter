<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser DOM ← User input | Item name strings, price strings (free-text), and chip-tap selections. All processed client-side. No network calls. |
| Browser ← In-memory state (Zustand) | `items[]`, `assignments` Record. No persistence in Phase 1. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-09 | Tampering | `Item.name` rendered into DOM via React JSX | mitigate | React renders strings as text nodes — no `dangerouslySetInnerHTML`. Cap input `maxLength={100}` on the name `<Input>` to prevent oversized DOM nodes. |
| T-01-10 | Denial of Service | Price `<Input>` accepting unbounded digits before validation | mitigate | Set `maxLength={9}` on price input (`9999999.99` ceiling); regex validates `/^\d+(\.\d{1,2})?$/` before `Math.round`. Reject before storing. |
| T-01-11 | Tampering | `assignments` map keyed by ItemId from `crypto.randomUUID()` | mitigate | All ItemIds and PersonIds are generated server-side (in-store) by `crypto.randomUUID()`. UI components only pass IDs they read from the store; no path for user-supplied keys to enter the assignments map. No prototype-pollution surface. |
| T-01-12 | Tampering | `setAssignment` overwriting prior assignment via `[...current, personId]` | mitigate | Toggle logic uses array filter/spread on PersonIds the UI already trusts (came from `people[]`). No path for arbitrary IDs to be inserted. |
| T-01-13 | Information Disclosure | Item names + prices visible to anyone with physical access to the device | accept | Phase 1 is single-device, single-session. No multi-user disclosure path. Phase 4 will introduce a sharing surface and re-evaluate. |
| T-01-14 | Denial of Service | Adding 1,000+ items via repeated `addItem` calls | accept | Restaurant-bill domain rules out adversarial volume. No real-world threat model. Plan 5 polish may add a soft cap if needed. |
| T-01-15 | Repudiation | No audit log of item or assignment changes | accept | Single-user, single-session — no accountability requirement in Phase 1. |
</threat_model>

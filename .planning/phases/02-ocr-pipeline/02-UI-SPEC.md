## Phase 2 Additions

Phase 2 adds four new UI surfaces to Step 2 (AddItems). All surfaces use the inherited design system — no new tokens, no new font sizes, no new palette values.

### Surface A — "Scan bill" Button

**Location:** Step 2 (AddItems), above the items list, below the step heading.

**Appearance:**
- `variant="outline"` shadcn Button (secondary, not accent-filled — scan is optional, not the primary path)
- Height: 48px (touch target requirement)
- Width: full width (`w-full`)
- Icon: `Camera` (lucide-react) left of label, 20px, `gap-2`
- Label: "Scan bill"
- Border: `border-zinc-300 border-dashed` (matches the existing "Add item" dashed affordance pattern)
- Text color: `text-zinc-600` at rest
- On press: `hover:bg-zinc-50` (consistent with existing Add item row)

**Placement rules:**
- Shown when `ocrStatus === 'idle'` (before any scan attempt)
- Hidden when `ocrStatus === 'loading'` (overlay is active)
- Hidden after scan succeeds (`ocrStatus === 'done'`) — replaced by thumbnail (Surface B)
- Re-shown after scan failure (`ocrStatus === 'error'`) — user can retry

**Underlying input element:**
- Rendered as `<input type="file" accept="image/*" capture="environment" className="sr-only" />` paired with a visible `<Button>` that triggers the input's click
- The `<input>` is visually hidden with `sr-only`; the Button carries all visible affordance
- `aria-label="Scan bill"` on the Button (icon + text label makes this redundant, but include for clarity)

**Position in Step 2 layout (top to bottom):**
1. Step heading ("Items" / empty state heading)
2. Bill thumbnail (Surface B) — appears after capture
3. "Scan bill" button (Surface A) — shown unless scanning or already successful with no retry
4. Items list (existing)
5. "Add item" dashed row (existing)
6. "Assign items" CTA (existing, amber-600)

---

### Surface B — Bill Thumbnail

**Location:** Step 2 (AddItems), between step heading and "Scan bill" button / items list.

**Appearance:**
- Container: shadcn `Card` with `p-2` padding, `rounded-xl`, `border border-zinc-200`
- Thumbnail image: `<img>` with `rounded-lg object-cover w-full` and a fixed height of `96px`
- Below the image: a single label line at 14px / weight 400 / `text-zinc-500`: "Bill photo"
- No delete/retake button on the thumbnail in Phase 2 — retaking is done by tapping "Scan bill" again (the button re-appears on error; a new scan in idle state overwrites the existing thumbnail)

**State rules:**
- Hidden when `billImageUrl === null` (initial state, or before any scan)
- Shown when `billImageUrl` is set (after successful scan OR on error if image was already set before error)
- The thumbnail persists for the full session once set (stored in Zustand as `billImageUrl: string | null`)
- `billImageUrl` is a blob URL (`URL.createObjectURL(file)`) — not a data URL

**Accessibility:**
- `<img>` element carries `alt="Captured bill photo"`
- Container Card has no role override needed (presentational)

---

### Surface C — Full-Screen Loading Overlay

**Location:** Rendered as a sibling of WizardShell content — full viewport cover, not scoped to Step 2.

**Behavior (source: D-06, CONTEXT.md):** Blocks all interaction during OCR processing. Dismissed only when OCR completes (success or failure).

**Appearance:**
- Background: `fixed inset-0 z-50 bg-zinc-950/80` (80% opaque dark overlay — readable in restaurant lighting)
- Center content: vertically and horizontally centered (`flex items-center justify-center`)
- Spinner: `LoaderCircle` lucide-react icon, 40px, `text-white`, `animate-spin`
- Message: "Scanning your bill…" — 16px, weight 400, `text-white`, `mt-4` below spinner
- No dismiss button — the overlay is non-dismissible while loading

**Implementation note:** Use a CSS `fixed` div rendered via a React portal (or at root layout level), NOT a shadcn Dialog. Dialog introduces focus trap and ARIA roles that conflict with this use case (it is not a dialog, it is a blocking process state). Render via `ReactDOM.createPortal` targeting `document.body`.

**Trigger:** `ocrStatus === 'loading'`

**Dismiss:** When `ocrStatus` transitions from `'loading'` to `'done'` or `'error'` — overlay unmounts.

**Animation:** Overlay fades in over 150ms (`transition-opacity duration-150`) on mount. No fade-out — dismiss is instant to feel responsive.

---

### Surface D — Error Toast

**Location:** Bottom of viewport, above safe area inset.

**Behavior (source: D-07, CONTEXT.md):** Shown on OCR failure (timeout, bad image, API error). Auto-dismisses after 4 seconds. Does not prevent interaction — user is back in Step 2 with manual entry available immediately.

**Appearance:**
- Implementation: `@base-ui/react` Toast primitive (already in `node_modules/@base-ui/react/toast`, no install needed)
- Container: `fixed bottom-4 left-4 right-4 z-50` (within safe-area), `max-w-[480px] mx-auto`
- Toast card: `bg-zinc-900 text-white rounded-xl px-4 py-3 shadow-lg`
- Message: "Couldn't read the bill — try again or enter manually"
  - 16px, weight 400, `text-white`
- No action button on the toast — recovery instruction is inline in the copy
- Auto-dismiss: 4000ms timeout
- No close/X button (auto-dismiss at 4s is sufficient; keep the surface clean)

**Trigger:** `ocrStatus === 'error'`

**Accessibility:**
- Toast container must carry `role="status"` and `aria-live="polite"` (Base UI Toast handles this)
- Do not use `aria-live="assertive"` — the error is not a crisis, and assertive interrupts screen reader users mid-flow

## Updated Step 2 Interaction Contract

This section extends (not replaces) the Phase 1 Step 2 interaction contract.

### New interactions added to Step 2

| Trigger | Outcome |
|---------|---------|
| Tap "Scan bill" button | Opens native file picker (`<input type="file" capture="environment">`); user selects/takes photo |
| File selected | `ocrStatus` → `'loading'`; overlay (Surface C) mounts; image compressed to ~500KB JPEG via `browser-image-compression`; POST to `/api/ocr`; thumbnail set immediately from blob URL |
| OCR succeeds | `ocrStatus` → `'done'`; overlay unmounts; extracted items batch-inserted via `addItem()` loop; items appear inline in the existing editable list; "Scan bill" button hides; thumbnail stays |
| OCR fails | `ocrStatus` → `'error'`; overlay unmounts; error toast (Surface D) shows for 4s; "Scan bill" button re-appears; items list unchanged (no partial items added on failure) |
| File picker cancelled (no file selected) | No state change — overlay never mounts, `ocrStatus` stays `'idle'` |

### Step 2 layout when scan is complete

```
[Step heading: "Items"]
[Bill thumbnail — 96px tall card]     ← new (Surface B)
[Items list — OCR items + any manual items, all editable]
[+ Add item (dashed row)]
[Assign items →]                       ← amber-600, enabled when ≥1 item
```

### Step 2 layout when scan has not been done yet (initial)

```
[Empty state heading: "What did everyone order?"]
[Empty state body text]
[Scan bill button — dashed outline]    ← new (Surface A)
[+ Add item (dashed row)]
[Assign items →]                       ← disabled, no items yet
```

### Step 2 layout when items exist (no scan or after scan with items)

```
[Step heading: "Items"]
[Bill thumbnail — if scan was done]    ← conditional (Surface B)
[Scan bill button — if ocrStatus !== 'done']  ← conditional (Surface A)
[Items list]
[+ Add item (dashed row)]
[Assign items →]
```

### Manual entry stays intact (D-04, D-05)

The "Add item" dashed row remains visible at all times in Step 2, below the OCR-populated list. User can add items manually before, during (not possible — overlay blocks), and after a scan. OCR items and manually-added items appear in the same flat list, all editable via the existing inline-edit pattern.

## Color

Inherited from Phase 1. No new palette values.

| Role | Value | Phase 2 usage |
|------|-------|---------------|
| Dominant (60%) | `#ffffff` / `#09090b` zinc-950 | Page background; overlay background is `zinc-950/80` (opacity variant) |
| Secondary (30%) | `#f4f4f5` zinc-100 / `#18181b` zinc-900 | Thumbnail card border; error toast background (`zinc-900`) |
| Accent (10%) | `#d97706` amber-600 | "Assign items" CTA (unchanged); progress strip (unchanged) |
| Destructive | `#dc2626` red-600 | Remove item Dialog (unchanged) |

**Overlay background:** `bg-zinc-950/80` — uses Tailwind opacity modifier. This is not a new color token; it is an opacity variant of the existing dominant dark value.

**Toast background:** `bg-zinc-900` — uses existing Secondary dark value. Toast is dark even in light mode (contrast + restaurant environment readability).

Accent is NOT used for the "Scan bill" button. Scan is optional; using amber on it would compete with the "Assign items" primary CTA. The dashed outline matches the established "Add item" affordance instead.

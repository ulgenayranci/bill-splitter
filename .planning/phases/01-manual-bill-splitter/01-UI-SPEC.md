## Flow Layout

Phase 1 uses a **single-page wizard** with 5 steps. Step navigation is linear; the user moves forward with a primary CTA and backward with a back button. No tabs, no sidebar, no persistent nav. Steps:

| Step | Route segment | Requirement |
|------|---------------|-------------|
| 1. Add People | `/` (step=1) | PEOPLE-01 |
| 2. Add Items | step=2 | ITEMS-01 |
| 3. Assign Items | step=3 | ITEMS-02, ITEMS-03 |
| 4. Set Tip | step=4 | TIP-01 |
| 5. Results | step=5 | RESULTS-01 |

Step state lives in the Zustand store (`step` field). URL hash (`#step-2`) updates on each advance so back-button works as expected on mobile.

**Rationale (Claude's discretion — CONTEXT.md flow layout):** A bottom-sheet or full-page step wizard is the correct pattern for a mobile-first sequential flow at a restaurant table. Multi-column layouts, tabs, and sidebars are inappropriate at 390px viewport width under dim lighting. Each step gets the full viewport. Source: CONTEXT.md "Flow layout — Claude decides based on mobile UX best practices."

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge padding |
| sm | 8px | Inline padding inside chips, compact inputs |
| md | 16px | Default element spacing, form field gap, list item padding |
| lg | 24px | Section padding, card padding |
| xl | 32px | Step header bottom margin, major layout gaps |
| 2xl | 48px | Between wizard steps, screen vertical breathing room |
| 3xl | 64px | Page-level bottom padding (clear of mobile browser chrome) |

Exceptions:
- Touch targets: minimum 48px height on all tappable elements (exceeds 44px iOS HIG minimum — chosen for restaurant dim-light usability per PITFALLS.md "large touch targets")
- Person color avatar: 40px diameter (outside the grid but constrained by circular avatar convention)
- Step progress indicator: 4px height strip at top of screen (intentionally below xs — decorative only, not interactive)

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 (regular) | 1.5 |
| Label | 14px | 400 (regular) | 1.4 |
| Heading | 20px | 600 (semibold) | 1.2 |
| Display | 28px | 600 (semibold) | 1.1 |

**Usage guidance:**
- Display: per-person totals on the Results screen only. Large enough to read at arm's length across a table.
- Heading: step titles ("Add people to your bill"), section headers within a step ("Items")
- Body: item names, person names in lists, tip descriptions
- Label: input field labels, secondary metadata (price sub-labels, "shared" badges, person count under shared item)

**Font size note:** No size outside the 4 declared values above. In particular: do not use 12px anywhere — unreadable under restaurant lighting. Do not use 24px — the jump from 20 to 28 is intentional to make the results display prominent.

## Color

**Source:** shadcn default neutral palette (zinc scale). No custom brand palette needed for Phase 1 — the product's neutrality is intentional. Accent is a single warm amber to connote "money / receipt."

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` (light) / `#09090b` zinc-950 (dark) | Page background, wizard step background |
| Secondary (30%) | `#f4f4f5` zinc-100 (light) / `#18181b` zinc-900 (dark) | Cards (item cards, person cards), input fields, step progress track |
| Accent (10%) | `#d97706` amber-600 | Primary CTA buttons, active wizard step indicator, selected person avatar ring, "total owed" amount text on Results screen |
| Destructive | `#dc2626` red-600 | Remove person, remove item, clear bill actions — destructive confirm dialogs only |

Accent reserved for: **primary CTA button fill, active step dot in progress indicator, selected-person highlight ring on item assignment cards, final "you owe" dollar amounts on Results screen.**

Not accent: secondary buttons, nav back arrows, form fields, unselected states, informational text. The amber is scarce — it draws the eye to the action that moves the user forward.

**Dark mode:** Implemented via shadcn's CSS variable system (`prefers-color-scheme`). The zinc scale inverts naturally. Amber-600 is readable on both zinc-950 and zinc-100 backgrounds at the stated WCAG contrast level.

## Interaction Contract by Step

### Step 1 — Add People

**Layout:** Vertical list of added people + an inline text field + "Add" button pinned to the field.

**Interactions:**
- Type name → tap "Add Person" or press Return → person appears in list with a colored avatar circle
- Avatar colors assigned deterministically: cycle through 6 preset colors (`amber-400, sky-400, emerald-400, violet-400, rose-400, orange-400`) in insertion order — never random
- Tap person row → no action (non-interactive in Phase 1)
- Tap trash icon on person row → Dialog confirmation: "Remove [Name]? Any items assigned to them will be unassigned." → Confirm removes person; Cancel dismisses. **Trash icon must carry `aria-label="Remove [Name]"` (where [Name] is the actual person's name) — it is icon-only with no visible text label.**
- Minimum 1 person required before CTA is enabled
- CTA: "Continue to items" (disabled / `opacity-50 cursor-not-allowed` until ≥1 person added)
- Empty state: heading "Who's at the table?", body "Add each person's name. You'll assign items to them next." — full viewport centered

**Touch targets:** Person row height 56px. Trash icon tap zone 48×48px. "Add Person" button 48px height.

### Step 2 — Add Items

**Layout:** Scrollable list of added items + a pinned "Add item" row at bottom of list.

**Interactions:**
- Tap "Add item" → a new inline row appears with name field focused, price field to the right
- Name field: free text. Price field: numeric keyboard (`inputMode="decimal"`), no currency symbol in the field — formatted with "$" prefix in display-only view after committing
- Commit item: tap away, press Return on price field, or tap checkmark button — item enters list
- Item in list shows: item name (body, 16px), price right-aligned (label, 14px)
- Tap item row → enter inline edit mode for that row
- Tap trash on item → Dialog: "Remove [item name]? It will be unassigned." → Confirm. **Trash icon must carry `aria-label="Remove [item name]"` (where [item name] is the actual item name) — it is icon-only with no visible text label.**
- Minimum 1 item required before CTA is enabled
- CTA: "Assign items" (disabled until ≥1 item)
- Empty state: heading "What did everyone order?", body "Add each item from the bill. You'll split them next."

**Price input rules:**
- Accept only digits and one decimal point
- Store as integer cents immediately on commit: `Math.round(parseFloat(value) * 100)`
- Display always shows 2 decimal places: `(cents / 100).toFixed(2)` — display only, never used in calculations
- If price is empty on commit, reject with inline red border + label "Enter a price"

### Step 3 — Assign Items

**Layout:** Scrollable list of items, each as a Card with person-assignment controls inline.

**Interactions:**
- Each item card shows: item name (body), price (label, right-aligned)
- Below the item: horizontal row of person avatar chips (tappable)
- Tap one person chip → item assigned solely to that person (chip fills with person's color, amber ring)
- Tap a second person chip while one is already selected → shared mode activates: both chips fill, item shows "Shared" badge
- Tap a filled chip to de-select that person from shared assignment; if only 1 person remains, badge disappears
- A person chip tapped alone while they are the sole assignee → de-selects (item becomes unassigned)
- Unassigned item: shown with a neutral gray border, no filled chips — valid to proceed (Phase 5 adds warning for this; Phase 1 allows it silently)
- CTA: "Set tip" — enabled regardless of assignment state (Phase 5 adds unassigned warning guard)

**Person chip size:** 40px diameter circle with 2-letter initials or single letter. 8px gap between chips. Row wraps if >5 people.

**Shared item math note (visible to user):** When 2+ people share an item, the Card shows a secondary line: "Split equally — $X.XX each" where X.XX is `price / sharers` formatted to 2dp. This is display only (computed from cents, not stored).

### Step 4 — Set Tip

**Layout:** Single centered card. Tip preset buttons + custom amount input.

**Interactions:**
- Four Button variants side by side: "15%", "18%", "20%", "Custom"
- Tapping a preset selects it (filled/accent variant), deselects others
- "Custom" tap reveals an Input field for percent entry (numeric, `inputMode="decimal"`) with "%" suffix — enter any non-negative value
- Default selection on step entry: 18% (most common US restaurant tip)
- Below buttons: a single line showing computed tip amount in dollars: "Tip: $X.XX" — updates live as selection changes, computed from current items' total cents
- CTA: "See results" — always enabled (tip=0 is a valid choice; user can tap "0" in custom or simply pick 0)

**Tip math (display only):**
- Total tip cents = `Math.round(subtotalCents * (tipPercent / 100))`
- Display: `(tipCents / 100).toFixed(2)`

**Equal split note (D-02):** In Phase 1, tip is split equally among all people. The step does not display per-person tip amounts here — that is deferred to Step 5 (Results).

### Step 5 — Results

**Layout:** List of person result cards, each collapsible. Fixed bottom strip showing total bill amount.

**Interactions:**
- Each card shows: person avatar + name (heading, 20px), total owed in Display type (28px, weight 600, amber-600)
- Tap card → expands below the total to reveal: itemized list (item name + their share amount, both 14px), then tip share ("Tip: $X.XX"), then a line total
- Expanded state: chevron rotates 180°; card grows in-place, pushing cards below down (no modal/sheet)
- Only one card can be expanded at a time (collapse previous on new tap) — reduces cognitive load
- Bottom strip (fixed, zinc-100/zinc-900 background): "Total bill: $X.XX" — sum of all items + full tip amount formatted to 2dp
- No CTA on this screen in Phase 1 — this is the terminal screen
- Back button returns to Step 4 (Tip) — user can adjust tip and see results update

**Results math (D-03, source: CONTEXT.md):** Default view shows per-person total only. Tap reveals breakdown. This satisfies the trust requirement from PITFALLS.md Pitfall 10 without overwhelming the at-a-glance view.

## Mobile-First Constraints

These rules apply to every screen in Phase 1 (source: PITFALLS.md, CONTEXT.md specifics):

1. **Viewport assumption:** 390px width (iPhone 14/15 base). No responsive breakpoints needed — the wizard is a single-column flow on all screen sizes. Max content width: 480px, centered on tablet.
2. **Touch targets:** 48px minimum height on all tappable elements. No exceptions for icon-only buttons.
3. **Font sizes:** Minimum 14px (Label). No 12px. No subpixel sizes.
4. **Bottom chrome clearance:** All fixed bottom elements use `padding-bottom: env(safe-area-inset-bottom, 16px)` to clear iOS home indicator.
5. **Keyboard avoidance:** Price and name inputs are never positioned in the bottom half of the screen — place them at top of the form. On iOS, the virtual keyboard pushes the viewport up; inputs in the lower half become partially obscured.
6. **No hover states as primary affordance:** All interactive state changes must work on tap alone. Hover effects are additive only.
7. **No tooltips:** Dim restaurant lighting + mobile = tooltips never work. If a UI element needs explanation, it needs a visible label.

## Copywriting Contract

### Per-step CTAs

| Step | CTA Label | Disabled State Label |
|------|-----------|---------------------|
| 1 — Add People | "Continue to items" | "Add at least one person" (disabled, shown as tooltip-free placeholder text below button) |
| 2 — Add Items | "Assign items" | "Add at least one item" |
| 3 — Assign Items | "Set tip" | (always enabled — no guard in Phase 1) |
| 4 — Set Tip | "See results" | (always enabled) |
| 5 — Results | (none — terminal step) | — |

### Empty States

| Screen | Heading | Body |
|--------|---------|------|
| Step 1 — no people yet | "Who's at the table?" | "Add each person's name. You'll assign items to them next." |
| Step 2 — no items yet | "What did everyone order?" | "Add each item and its price. You'll split them in the next step." |
| Step 3 — item with no assignees | (inline, no heading) | "Tap a name to assign" (shown inside the unassigned item card, muted text) |

### Error States

| Trigger | Message | Recovery |
|---------|---------|---------|
| Price field submitted empty | "Enter a price" | Inline red border + label beneath the price field; field retains focus |
| Price field contains non-numeric text | "Numbers only" | Inline; field clears invalid characters on blur |
| Duplicate person name | "You already added [Name]" | Inline below name input; does not block adding (names are not unique — two people can share a name) — this is informational, not blocking |

**Note:** Duplicate names are allowed (two people named "Alex" at a table is real). The warning is informational only.

### Destructive Actions

| Action | Trigger | Confirmation Copy |
|--------|---------|------------------|
| Remove person | Trash icon on person row | Dialog: "Remove [Name]?" / body: "Any items currently assigned to them will become unassigned." / buttons: "Remove" (destructive red) + "Cancel" |
| Remove item | Trash icon on item row | Dialog: "Remove [item name]?" / body: "It will be removed from all assignments." / buttons: "Remove" (destructive red) + "Cancel" |
| Start over | (Phase 1: not exposed in UI — no "clear bill" button. Deferred: browser refresh is the escape hatch.) | — |

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser localStorage → identity | Stored personId is client-controlled; used to restore identity for this session |
| client → /claim (share) & /edit (add_person) | Untrusted bodies cross to Redis; validated/atomic server-side (Plans 01, 02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-14 | Spoofing | localStorage identity spoofing within a session | mitigate | Restore only honors a stored personId whose slot is present in server `personSlots`; an arbitrary forged id fails the check and re-prompts the modal. Attribution is per the slot the server locked |
| T-09-15 | Tampering | claim/edit races (concurrent devices) | mitigate | All writes go through atomic Lua (share + add_person); SWR optimistic updates use rollbackOnError so a rejected write reverts the UI |
| T-09-16 | DoS | spam "I'm not listed" / share-spam | mitigate | add_person capped at 20 (Plan 02); share writes idempotent (Plan 01); anonymous 24h-TTL session, low value |
| T-09-17 | Information Disclosure | share-link exposure via header/dialog | accept | Sharing the join link is the intended capability (CLAIM-06); no auth, no PII beyond names |
| T-09-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages (RESEARCH audit: none); all components pre-built in Plans 03-05 |
</threat_model>

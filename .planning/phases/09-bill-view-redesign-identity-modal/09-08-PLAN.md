<objective>
Close GAP-09-NOLOCK: remove name-locking entirely from the identity flow. In the flat collaborative model there is NO host role, so exclusive ownership of a name is the wrong model. Every name in the "Who are you?" modal must ALWAYS be re-selectable by anyone, two devices may be the same person at once and co-edit that person's share, and claiming must work without holding an exclusive lock.

This OVERRIDES decision D-01 ("taken names are greyed out") and the locking interpretation of requirement IDENT-02. It is a user-confirmed product-owner directive (2026-06-08). It is an accepted trade-off: the flat-model session link already grants full edit access to anyone (threat T-09-FLOW-02, already accepted), so removing the lock does not widen the access model.

Purpose: Make name selection unconstrained and concurrent same-name co-editing work, without breaking any of the 7 UAT behaviors that already pass (especially: the first-load modal must STILL block dismissal until a name is picked — that is separate from name-locking and stays).

Output: PersonSlotPicker renders all names selectable; CollaborativeClaimingView restores identity by person existence; the claim route drops the slot_taken guard and the qty/share ownership guards; the edit route stops setting an exclusive lock on add_person; the per-person ownership guards in done/tip are removed for consistency; and the affected test suites are inverted to the no-lock model.

## personSlots decision (STATED, per the gap directive)
`personSlots` is KEPT in the schema (`lib/sessionSchema.ts` Record<PersonId, boolean>) and may still be WRITTEN as a harmless presence marker (SLOT_CLAIM_SCRIPT still sets `personSlots[personId] = true`, ADD_PERSON still ensures the table exists). The single coherent rule applied everywhere: **stop treating `personSlots[id]===true` as an EXCLUSIVE lock.** Selection always succeeds (no slot_taken). Claiming never requires the lock (no qty/share 'forbidden'). done/tip stop gating on it. Identity restore is membership-based. We do NOT delete the field — that would be a larger schema change touching the writers in session/done/tip routes for no behavioral benefit; keeping it written-but-never-gated is the minimal correct fix.
</objective>

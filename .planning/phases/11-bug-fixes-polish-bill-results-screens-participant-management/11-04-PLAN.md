<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client UI → /edit route | remove/rename triggered from the people modal; personId + newName cross to the server (validated server-side in 11-01) |
| SWR poll → render decision | session.people drives whether the claiming view, identity modal, or expired screen renders |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-09 | Tampering | XSS via rename input | mitigate | newName rendered as a React text node (JSX escape) in cards/attribution; no dangerouslySetInnerHTML; server also trims/length-caps in 11-01 |
| T-11-10 | Spoofing | anyone removes/renames anyone from the modal | accept | INTENDED by D-07 / flat no-lock model — no auth; ephemeral session |
| T-11-11 | Denial of Service | self-removal stranding the viewer on SessionExpiredScreen | mitigate | self-removal useEffect re-opens the identity modal so the removed viewer can re-select/re-add, instead of a dead-end expired screen |
| T-11-12 | Tampering | stale editingPersonId after concurrent removal | mitigate | PersonSlotPicker useEffect clears editingPersonId when the person leaves session.people (Pitfall 4) |
| T-11-13 | Information Disclosure | person_not_found on concurrent double-remove | accept | treated as soft success client-side (Pitfall 5); end-state already achieved, no error surfaced |

Note: No new auth requirements are introduced — the flat model (D-07) intentionally lets any participant manage any participant on an ephemeral 24h-TTL session.
</threat_model>

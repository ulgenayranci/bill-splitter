<objective>
Add two new atomic Lua ops to the `/edit` route — `remove_person` and `rename_person` — implementing decisions D-05, D-06, and D-07. Removing a person purges their entire footprint (people[], claims.items[*][personId], personSlots, donePeople, tips) in a single Redis eval, freeing their claimed items back to unclaimed. Renaming updates the name atomically with server-side validation. A light `last_person` guard prevents a 0-person session.

Purpose: Participant management (remove + rename) has no server surface today. This plan builds the backend foundation that the Wave 2 UI plan (11-04) consumes. Server-first ordering means the UI plan can call real endpoints, not stubs.

Output: Extended `edit/route.ts` with two new ops + full test coverage in `editRoute.test.ts`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → /edit route | Untrusted JSON body (op, personId, newName) crosses into the server; validated by validateOp before any eval |
| /edit route → Redis (Lua eval) | Server-controlled KEYS[1] = `session:${sessionId}`; only ARGV (personId, newName) is client-derived |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01 | Tampering | remove_person / rename_person inputs | mitigate | validateOp rejects non-string/empty personId and non-string/empty/>50-char newName with 400 before eval (ASVS V5) |
| T-11-02 | Tampering | cross-session person removal | mitigate | Lua uses server-built KEYS[1] = `session:${sessionId}`; client personId can only address persons within the caller's own session — cross-session access structurally impossible |
| T-11-03 | Spoofing | anyone removes/renames anyone | accept | INTENDED by D-07 / [[project_no_name_locking]] flat model — no auth in this app; ephemeral 24h-TTL session; mass-removal abuse out of scope |
| T-11-04 | Denial of Service | mass-removal dropping session to 0 | mitigate | last_person guard blocks removal when people length <= 1 (409), preventing a structurally-broken 0-person session |
| T-11-05 | Information Disclosure | person_not_found reveals absence | accept | Low risk — sessionId is already known to the caller; no PII leaked |
</threat_model>

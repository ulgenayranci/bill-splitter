## Flow Layout

### Host Path — Modified Wizard

The existing `WizardShell` progress strip updates to reflect the step order change. Steps 1–3 are unchanged visually. Step 4 gains a dual-mode layout.

**Step 4 — AssignItems/Share (modified):**

- Retains all existing manual assignment UI unchanged above the fold
- Gains a prominent "Share link" button in the bottom CTA row, positioned alongside the existing "See results" CTA
- CTA row layout: `[See results (outline, flex-1)] [Share link (amber-600 filled, flex-1)]`
- Tapping "Share link" triggers session creation, then transitions the wizard to a new waiting sub-state (step 5a within the wizard shell)

**Step 5a — Host Waiting Screen (new):**

- Replaces the standard Results screen when the sharing path was chosen
- Full-screen within the `WizardShell` max-width container
- Not a separate route — same `/` page, wizard step 5 with a `hostMode: 'waiting'` flag in store
- Has its own "View results" CTA that transitions to the standard Results screen (step 5) once the host chooses to stop waiting

### Guest Path — New Route

`/split/[sessionId]` is a standalone page, not inside the wizard shell. It has its own layout: single column, max-width 480px centered, no wizard progress strip.

Inherits the 8-point scale established in Phase 1. No new exceptions for Phase 4.

| Token | Value | Usage |

## Color

Inherits the amber/zinc palette established in Phase 1. Phase 4 adds one semantic treatment for claimed-item dimming.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `oklch(1 0 0)` — white / `oklch(0.145 0 0)` — zinc-950 dark | Page background for `/split/[sessionId]`, wizard step backgrounds |
| Secondary (30%) | `oklch(0.97 0 0)` — zinc-100 / `oklch(0.205 0 0)` — zinc-900 dark | Item cards on claiming page, person rows on waiting screen, bottom CTA bar |
| Accent (10%) | `#d97706` amber-600 | "Share link" primary CTA fill, "I'm done" primary CTA fill, guest's claimed item checkmark, guest total on done screen, active step in progress strip |
| Destructive | `oklch(0.577 0.245 27.325)` — red-600 | Not used in Phase 4 (no destructive actions introduced) |

Accent reserved for: **"Share link" button fill on step 4, "I'm done" button fill on guest claiming page, checkmark icon on items the current guest has claimed, and the final personal total amount on the guest done screen.**

**Claimed-item dimming:** Items claimed by another person use `opacity-50` on the card combined with `bg-muted` — no new color introduced. The "Taken by [Name]" label uses `text-muted-foreground` (zinc-500). The avatar color dot next to "Taken by [Name]" uses the same `AVATAR_COLORS` array as the rest of the app (source: D-08 + CONTEXT.md specifics).

**Waiting screen progress:** Person rows use colored avatar circles from `AVATAR_COLORS`. Done state: green checkmark (`text-emerald-500`). Pending state: animated spinner (`text-zinc-400`, `animate-spin`). This is the only place green appears — it is not a new accent; it is a status indicator distinct from amber.

## Interaction Contract by Screen

### Modified: Step 4 — AssignItems/Share

The existing AssignItemsStep manual assignment UI (item cards + person avatar chips) is unchanged above the fold. The only modification is to the bottom CTA row.

**Updated CTA row:**
- Left button: "See results" (outline variant, `flex-1`, 48px height) — navigates to manual Results step (existing behavior)
- Right button: "Share link" (amber-600 filled, `flex-1`, 48px height) — triggers the sharing flow

**"Share link" tap sequence:**
1. Button shows loading state: spinner replaces label text, button disabled (`cursor-not-allowed`)
2. POST to `/api/session` with serialized bill state (people, items, assignments, tipPercent)
3. On success: transition wizard to step 5 waiting mode; URL is available for copying
4. On failure: toast error "Couldn't create session. Try again." — button returns to enabled state

**Touch targets:** Both CTA buttons 48px height minimum (established constraint).

---

### New: Host Waiting Screen (Wizard Step 5 — Sharing Mode)

**Layout:** Single column within WizardShell max-width container. Three sections stacked vertically:
1. Share URL banner (top)
2. Person progress list (middle, scrollable)
3. "View results" CTA (bottom, pinned)

**Share URL banner:**
- Display: truncated session URL with a copy icon button
- Copy button tap: writes full URL to clipboard, shows brief "Copied!" confirmation text inline (replaces copy icon for 2 seconds, then reverts) — no toast needed
- The URL itself: `[origin]/split/[sessionId]` — displayed in a `Card` with monospace-ish body font

**Person progress list:**
- One row per person in `people[]` array from session
- Each row: avatar circle (40px, `AVATAR_COLORS[person.colorIndex]`) + person name (body, 16px) + status indicator
- Status indicator states:
  - Waiting (not yet opened link): animated spinner (`LoaderCircle` lucide, 16px, `text-zinc-400 animate-spin`)
  - Claiming (opened link, not done): dots animation or static "Claiming…" label (14px, muted)
  - Done: green checkmark (`Check` lucide, 16px, `text-emerald-500`)
- Polling: every 3 seconds (D-07), using `useEffect` + `setInterval`; no visual refresh indicator needed at this cadence

**"All done" banner:** When every person slot shows Done status, an "All done!" banner appears above the person list in green (`bg-emerald-50 text-emerald-700 border border-emerald-200`, 12px border-radius). Copy: "Everyone has claimed their items."

**"View results" CTA:**
- Always visible and enabled — host can choose to end waiting at any time
- Pinned to bottom using `fixed bottom-0` pattern from existing ResultsStep
- Background: `bg-background` with top border `border-t border-border`
- Padding: `p-4` + `env(safe-area-inset-bottom, 16px)`
- Button: full width, amber-600 filled, 48px height, label "View results"

### New: Guest Claiming Page — Identity Selection

Route: `/split/[sessionId]` (initial state before slot is claimed)

**Layout:** Full-page, no wizard shell. Single column, `max-w-[480px] mx-auto px-6 py-8`.

**Page structure:**
1. Heading: "Who are you?" (20px, semibold)
2. Body: "Pick your name from the list below." (16px, regular, muted)
3. Name grid: 2-column grid of person name buttons
4. No CTA — tapping a name is the action

**Person name buttons:**
- Each: `Card` with avatar circle (40px) + person name (body, 16px), centered
- Card dimensions: full-width of grid column, 72px minimum height (large touch target at a restaurant table)
- Available slot: normal Card styling, tappable
- Already-claimed slot: `opacity-50`, `cursor-not-allowed`, label "(taken)" appended in muted text (14px)
- Tap available slot: atomic claim attempt via POST `/api/session/[sessionId]/claim` for person slot; on success, transition to claiming view; on race-condition conflict (slot taken between render and tap), refetch and show slot as "(taken)"

### New: Guest Claiming Page — Item Claiming

Route: `/split/[sessionId]` (after slot claimed)

**Layout:** Three sections stacked:
1. Fixed top header: person name + avatar
2. Scrollable item list
3. Fixed bottom CTA: "I'm done"

**Fixed top header:**
- Height: 56px
- Background: `bg-background` + `border-b border-border`
- Content: avatar circle (32px, person's color) + "Hi, [Name]!" (heading, 20px, semibold), left-aligned with `px-6`
- Stays fixed while item list scrolls (`sticky top-0 z-10`)

**Scrollable item list:**
- `pb-[80px]` bottom padding to clear the fixed "I'm done" bar
- Each item: `ClaimableItemCard`

**ClaimableItemCard — states:**

| State | Visual Treatment |
|-------|-----------------|
| Unclaimed | Normal Card, full opacity. Checkbox-style indicator: empty circle outline (24px) on left. Item name (16px, body). Price (14px, label, right-aligned). |
| Claimed by me | Card background: `bg-amber-50` (subtle warm fill). Filled circle with checkmark (`Check` lucide, 16px, amber-600). Item name (16px, semibold). Price (14px, right-aligned). |
| Taken by other | Card: `opacity-50`. Gray filled circle. Item name (16px, regular, strikethrough — `line-through`). "Taken by [Name]" badge using `AVATAR_COLORS` colored dot + name (14px, muted). Price visible but muted. |

**Tap behavior:**
- Unclaimed → tap → POST claim → optimistic UI update to "Claimed by me" → confirm on poll response
- Claimed by me → tap → POST un-claim (D-09) → optimistic UI update to "Unclaimed"
- Taken by other → tap → no action; card is not interactive (`pointer-events-none`)

**Polling:** Every 3 seconds. Items update from poll response. Optimistic updates persist until poll overrides.

**Fixed "I'm done" bar:**
- `fixed bottom-0 left-0 right-0`
- Background: `bg-background border-t border-border`
- Padding: `px-6 py-4` + `env(safe-area-inset-bottom, 16px)` bottom
- Button: full width, amber-600 filled, 48px height, label "I'm done"
- "I'm done" tap: POST `/api/session/[sessionId]/done` with personId; transition to GuestDoneScreen

### New: Guest Done Screen

Shown after "I'm done" is tapped.

**Layout:** Single column, `px-6 py-8`, no scroll needed for typical item counts.

**Structure:**
1. Heading: "You're done!" (20px, semibold)
2. Personal total (display): amount in amber-600, 28px, semibold — e.g. "$14.50"
3. Subtext: "Your share" (14px, muted)
4. Separator
5. Item breakdown list: each claimed item name (14px) + share amount (14px, right-aligned)
6. Tip line: "Tip" (14px) + tip share amount (14px, right-aligned)
7. Separator
8. Total line: "Total" (14px, semibold) + total amount (14px, semibold)

No "Back" button — done is final. No navigation away from this screen (user closes browser tab to exit).

Math: all amounts computed from session state using the same `computePersonTotals` / largest-remainder logic as the existing ResultsStep.

---

### New: Session Expired / Not Found Screen

Shown when `/split/[sessionId]` cannot load a valid session from Redis (404, expired TTL, or network failure on first load).

**Layout:** Full-page centered vertically and horizontally. `flex items-center justify-center min-h-screen`.

**Content:**
- Icon: `LinkOff` (lucide, 48px, zinc-400)
- Heading: "This session has expired" (20px, semibold)
- Body: "The link you opened is no longer active. Ask the person who shared it to create a new one." (16px, regular, muted, centered)
- No button (no recovery action possible — user must get a new link)

Source: CONTEXT.md specifics — "No dead ends: if the Redis session expires or is not found, the `/split/[sessionId]` page should show a clear 'This session has expired' message, not a blank screen."

## Mobile-First Constraints

All Phase 1 mobile constraints carry forward unchanged. Phase 4 additions:

1. **Guest page has no wizard shell** — the `/split/[sessionId]` page uses its own layout, not `WizardShell`. Max content width 480px centered remains.
2. **Fixed header + fixed CTA bar pattern** — same `env(safe-area-inset-bottom)` clearance as established in ResultsStep.
3. **Claiming list scroll** — item list uses `pb-[80px]` to prevent content from hiding behind the fixed "I'm done" bar.
4. **No hover states as primary affordance** — all state changes (claim/unclaim) respond to tap only.
5. **Optimistic UI required** — 3-second polling means taps must reflect immediately. Do not wait for server confirmation before updating card state.
6. **Session URL copy** — use `navigator.clipboard.writeText()` with a fallback for older iOS (select + copy) — no user-visible failure if clipboard is unavailable; silently skip the "Copied!" confirmation.

### Error States

| Trigger | Message | Recovery |
|---------|---------|---------|
| Session not found / expired | "This session has expired" | Body: "Ask the person who shared it to create a new one." No button. |
| Session creation failure (POST /api/session error) | Toast: "Couldn't create session. Try again." | "Share link" button re-enables; user can retry |
| Claim conflict (slot taken by race) | Inline: slot button updates to "(taken)" state | User picks another name; no toast |
| Poll failure (network error during claiming) | No visible error — polling retries silently; item states remain at last known state | Stale state visible; next successful poll corrects |
| "I'm done" POST failure | Toast: "Couldn't save. Try again." | "I'm done" button re-enables |

No destructive actions in Phase 4. Un-claiming an item (D-09) is reversible by design — no confirmation dialog needed.

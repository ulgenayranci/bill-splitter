<diagnostics>
<!-- Root causes already diagnosed during UAT triage. Executor: apply directly. -->

GAP 1 (progress strip not visible) — ROOT CAUSE:
In WizardShell.tsx the strip wrapper is `<div className="flex h-[3px] w-full gap-1 px-5 pt-2.5">`.
The wrapper is clamped to a 3px box BUT also has `pt-2.5` (10px top padding). The
inner segment bars are `flex-1` with no explicit height, so their content box
collapses to ~0 inside the over-constrained, padded 3px wrapper — the strip is in
the DOM (the existing test passes) but renders with no visible height. Fix: give
the bars a real height and stop clamping the wrapper to 3px. See Task 1.

GAP 6 (stale item-count chip on failed retake) — ROOT CAUSE:
`billScanned = items.length > 0` gates the "after scan" UI (thumbnail + "N items
found" badge). The empty-scan path (around line 108-117) and the error-scan catch
(around line 118-127) both call `setBillImage(null)` + `setOcrStatus('error')` but
never clear `items`, so a prior successful scan's items survive and the badge +
canContinue stay true. `setItems` exists on the store (verified:
stores/useBillStore.ts line 126 `setItems: (items) => set({ items })`). Fix: call
`setItems([])` on both failure paths.

GAP 7 (toast position) — ROOT CAUSE:
Empty/error feedback is emitted via `toastManager.add({...})` (base-ui Toast),
which renders at the configured viewport (bottom). The "in-progress" status the
user references is the `OcrLoadingOverlay` (a centered full-screen overlay) — so
"same level" means: surface the failure message inline, near the scan tile,
instead of the bottom toast. The scan tile already has an error sub-line
("Something went wrong — tap to try again") shown when `ocrStatus === 'error'`.
Fix: replace the bottom toast with an inline error message rendered directly
above/at the scan tile, driven by a specific error string. See Task 2.
</diagnostics>

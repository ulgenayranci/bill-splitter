## Spacing Scale

Declared values (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Avatar gap, inline icon spacing, claimant avatar overlap offset |
| sm | 8px | Between item list rows, gap inside badge, stepper button internal padding |
| md | 16px | Card horizontal padding, header horizontal padding, section inner padding |
| lg | 24px | Page horizontal gutter (matches existing `px-6`), section vertical gaps |
| xl | 32px | Between major page sections |
| 2xl | 48px | PersonTipScreen top spacing before total display |
| 3xl | 64px | Not used in this phase |

Exceptions:
- Minimum touch target: 44px height on all interactive elements (tappable item cards, stepper buttons, host panel rows). Source: existing `min-h-[44px]` on ClaimableItemCard.
- Sticky header: 56px height (`h-14`). Source: existing GuestClaimingView `h-14` header.
- Fixed bottom action bar: 64px total height including `pb-4` + safe area inset. Source: existing fixed done bar pattern.
- Avatar size in claimant list: 16x16px (`h-4 w-4`) for stacked multi-claimant display. 32x32px (`h-8 w-8`) for header identity avatar. 40x40px (`h-10 w-10`) for PersonSlotPicker grid cards.

## Color

| Role | Value | Usage | Source |
|------|-------|-------|--------|
| Dominant (60%) | `bg-background` / `oklch(1 0 0)` white | Page background, sticky header background, fixed bar background | globals.css + existing patterns |
| Secondary (30%) | `bg-card` / `oklch(1 0 0)` with border | Item cards, PersonSlotPicker grid cells, HostPanel sections, ReviewScreen flagged item cards | existing Card component |
| Accent (10%) | `amber-600` (#d97706) | See reserved list below | CONTEXT.md phase_summary + existing components |
| Destructive | `red-600` (#dc2626) | Reject edit request (host action), error inline copy only | existing error text pattern + D-11 |

Accent (`amber-600`) reserved for:
1. Primary CTA button fill: "I'm done", "Confirm tip", "Accept all", "Share link" — `bg-amber-600 hover:bg-amber-700`
2. Active/claimed item check icon: `text-amber-600` Check icon on ClaimableItemCard when mine=true
3. Claimed item card background tint: `bg-amber-50` when mine=true
4. Progress strip fill: `bg-amber-600` active segments in WizardShell
5. PersonResultsScreen total amount text: `text-amber-600` (matches existing GuestDoneScreen)
6. HostPanel badge count: `bg-amber-600 text-white` rounded pill showing pending count
7. Quantity stepper active unit indicator: `text-amber-600` for the stepper's current claimed count

Secondary semantic colors (not destructive, but distinct from neutral):
- `orange-400` — low/ambiguous confidence badge (inherited from Phase 3, not used in Phase 6 UI)
- AVATAR_COLORS array — `bg-amber-400`, `bg-sky-400`, `bg-emerald-400`, `bg-violet-400`, `bg-rose-400`, `bg-orange-400` — used only for avatar circles; never as interactive or status colors

Host-assigned item flag color: `bg-amber-50` border `border-amber-200` — visually distinct from self-claimed amber-50 (self-claimed has no explicit border). Source: design discretion per CONTEXT.md.

Disputed item pending state: `bg-zinc-50` with `text-zinc-400` muted text + spinner icon. Indicates "awaiting host resolution" without using destructive red.

## Interaction Contracts

### ClaimableItemCard — Phase 6 Behaviors

**qty = 1 items (default):**
- Single tap: claim (mine=true → amber-50 bg, Check icon) / unclaim (mine=false → neutral)
- Multiple claimants: show stacked avatar circles (16px circles, 4px overlap) below item name
- Host-assigned items: `bg-amber-50 border border-amber-200`, small "Assigned by host" label in 12px zinc-500
- Optimistic update: SWR bound mutate with `optimisticData`, `rollbackOnError: true`

**qty > 1 items:**
- Card shows qty label: "×{N}" next to price in 14px zinc-500
- Tap opens inline quantity stepper (not modal): "How many did you have?" with [-] count [+] buttons, range [0..item.quantity]
- Stepper buttons: 44px touch target, amber-600 for active/non-zero count display
- Zero = no claim; submits on every stepper change with 300ms debounce
- Remaining unclaimed units shown below name: "X of N claimed" in 14px zinc-400

**Claimant display:**
- Up to 3 avatar circles shown stacked horizontally, each 16px, using AVATAR_COLORS
- If more than 3 claimants: "+N" text in 14px zinc-500 after the avatars
- Tooltip/accessible name: "{name1}, {name2} also claimed this"

### HostPanel — Layout Decision

Use a **bottom sheet** (Radix DrawerPrimitive or shadcn Sheet with `side="bottom"`). Rationale: Mobile-first; full-width bottom sheet matches the existing fixed-bottom action bar pattern. Tabs for the 3 sections (Edit Requests | Unclaimed | Disputes) using simple tab strip, not accordion — all sections are equal priority.

Trigger: floating action button in the bottom-right corner of the claiming view, 56x56px, amber-600 fill, with a badge showing total pending count across all 3 sections. Only visible when `isHost === true`.

Sheet height: 70% of viewport (`h-[70vh]`), scrollable content within each tab.

### EditRequestForm — Layout Decision

Rendered as a full-screen bottom sheet (`side="bottom"`, `h-[90vh]`). 4 request type options as a segmented tab row at top. Fields rendered below based on type:
- `add`: name input + price input + quantity stepper
- `remove`: item picker (dropdown listing all current session items) + confirmation message
- `edit_name`: item picker + new name input
- `edit_price`: item picker + new price input

Submit button at bottom of sheet, full-width, amber-600.

### "I'm done" → Review → Tip flow (D-08, D-09)

1. Tap "I'm done" → POST /api/session/[id]/done → server sets `donePeople[personId]=true`
2. If any items in `claims.items` have `assignedBy: 'host'` for this person → show `PersonDoneReviewScreen`
3. If no host-assigned items → go directly to `PersonTipScreen`
4. Back from `PersonDoneReviewScreen` → return to claiming (full edit rights, `donePeople` flag cleared client-side, re-POST on next "I'm done")
5. Back from `PersonTipScreen` → return to `PersonDoneReviewScreen` (if applicable) or claiming

### Dispute Pending State

While `disputes[id].status === 'pending'` for an item this person disputed:
- Item in `PersonDoneReviewScreen` (if revisited) shows: spinner icon (lucide `Loader2` animate-spin, 16px zinc-400) + "Waiting for host..." in 14px zinc-400
- Person cannot tap Accept or Dispute again on the same item until resolved
- Resolution is detected on next SWR poll — no action required by the person

### Tip Stepper (PersonTipScreen)

- Default display: "0%" in 28px semibold
- Increment options: preset buttons "10%", "15%", "20%" + custom input
- Custom input: number input, 16px, labeled "Custom %", accepts integers only
- Real-time preview: "Your total: {formatCents(itemSubtotal + tipCents)}" updates as percentage changes
- Zero tip is valid — no warning, no friction
- "Confirm tip" submits, transitions to PersonResultsScreen

## Screen Flow

```
WizardShell (4 steps: AddPeople, AddItems, AssignItems, ResultsStep)
  └─ ResultsStep
       └─ ShareLinkButton → POST /api/session
            └─ router.push(`/split/${sessionId}?hostToken=${hostToken}`)

/split/[sessionId] page
  └─ CollaborativeClaimingView
       ├─ [no slot selected] PersonSlotPicker
       └─ [slot selected] Claiming view
            ├─ Sticky header: avatar + "Hi, {name}!" + [host: HostPanel trigger FAB]
            ├─ Item list: ClaimableItemCard ×N
            ├─ [host only] EditRequestForm trigger (pencil icon in each item row)
            └─ Fixed bottom bar: "I'm done" CTA
                 ↓
                 [host-assigned items?]
                 ├─ YES → PersonDoneReviewScreen
                 │    ├─ Accept all → PersonTipScreen
                 │    ├─ Dispute item → host panel flagged; item shows pending spinner
                 │    └─ Back → claiming view
                 └─ NO → PersonTipScreen
                           ├─ Confirm tip → PersonResultsScreen
                           └─ Back → PersonDoneReviewScreen or claiming
```

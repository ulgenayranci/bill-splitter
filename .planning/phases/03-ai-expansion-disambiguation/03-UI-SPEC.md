## Phase 3 Additions

Phase 3 adds four new UI surfaces to Step 2 (AddItems), all within the inherited design system. No new tokens, no new font sizes, no new palette values — except the orange review color documented below.

The two loading phases (OCR → Expand) must feel like a single coherent flow. The expansion overlay is visually identical to the OCR overlay but carries different copy, preserving the "one process" perception.

---

### Surface E — Expansion Loading Overlay

**What:** Second full-screen blocking overlay shown while `/api/expand` is running, after OCR completes. Immediately follows the OCR overlay — the user sees "Scanning…" overlay dismiss and "Expanding…" overlay mount in sequence with no gap.

**Decision:** Reuse `OcrLoadingOverlay` component with a prop override (or extract a `message` prop). Same visual treatment as OCR loading — full-screen portal, same backdrop, same spinner. A lighter inline spinner was considered and rejected: the list is not yet populated at this point, so there is no content to show beneath a spinner, and the full-screen overlay makes the two-phase loading feel like one coherent action.

**Appearance:** Identical to Phase 2 Surface C.
- Background: `fixed inset-0 z-50 bg-zinc-950/80`
- Spinner: `LoaderCircle` lucide-react, 40px, `text-white`, `animate-spin`
- Message: "Expanding names…" — 16px, weight 400, `text-white`, `mt-4` below spinner
- No dismiss button — non-dismissible while loading

**Trigger:** `expandStatus === 'loading'`

**Dismiss:** When `expandStatus` transitions from `'loading'` to `'done'` or `'error'` — overlay unmounts instantly (no fade-out, matching OCR overlay dismiss behavior).

**Implementation:** Extend `OcrLoadingOverlay` to accept a `message` prop (default: `"Scanning your bill…"`). Render `<OcrLoadingOverlay visible={expandStatus === 'loading'} message="Expanding names…" />`.

**Accessibility:**
- `role="status"` and `aria-live="polite"` on the overlay div (same as OCR overlay)
- `aria-label="Expanding item names"` on the overlay div

### Surface F — "Review" Badge on Item Rows

**What:** An orange badge rendered on item rows where the AI returned `confidence: 'low' | 'ambiguous'`. Tapping the row (or the badge) opens the disambiguation dialog (Surface G). Badge dismisses when the user saves any edit on that item.

**Position:** Inside the item row Card, between the item name and the price. The badge sits inline in the flex row: `name [Review] price [trash]`.

**Exact layout within item row:**
```
[name text — flex-1] [Review badge] [price — 14px zinc-500] [Trash2 button — 48×48]
```

**Badge styling:**
- Use the existing `Badge` component from `components/ui/badge.tsx`
- Apply custom classes to override to orange — do NOT use an existing variant (none map to orange)
- Exact classes: `bg-amber-100 text-amber-700 border border-amber-300 text-xs font-medium`
- Dimensions: badge `h-5` (20px), `px-2`, `rounded-full` — matches the existing badge base shape
- Label: "Review"
- Do NOT use red or amber-600 (accent). Orange family: amber-100 background + amber-700 text creates a warm, non-alarming flag that reads as "attention needed" without "error"

**Interaction:**
- The entire item row remains tappable (same `onClick` as before) — tapping anywhere on the row opens the disambiguation dialog for that item instead of the inline edit form
- The badge itself does not need a separate click handler — it is inside the tappable row
- Touch target of the row is already min-h-14 (56px) — no change needed

**State rules:**
- Badge visible when: `item.confidence === 'low' || item.confidence === 'ambiguous'` AND `item.reviewed !== true`
- Badge dismissed when: user saves any edit on the item (calls `updateItem()`) — set `item.reviewed = true` in the store action or derive dismissal from the save action itself
- Items with `confidence: 'high'` never show a badge

**Zustand store note:** The `items` array needs to carry `confidence: 'high' | 'low' | 'ambiguous'` and optionally `rawName: string` per item. Items added manually default to `confidence: 'high'`. The badge dismissal approach: `updateItem()` clears confidence to `'high'` (or sets `reviewed: true`) — planner to decide exact shape, but badge must be gone after any save.

### Surface G — Disambiguation Dialog

**What:** A bottom-anchored sheet-style Dialog that opens when a "Review" item row is tapped. Contains two large tap targets: "Type name" and "Take menu photo". No forced action — user can dismiss via the X button.

**Trigger:** Tapping any item row where the "Review" badge is visible.

**Dialog structure:**

```
[DialogHeader]
  [DialogTitle] "What's this item?"
  [DialogDescription] AI's best guess for the name (pre-filled, not editable here)

[Two large tap-target buttons — stacked vertically]
  [Button 1] "Type name"          ← opens inline edit within dialog
  [Button 2] "Take menu photo"    ← triggers camera, runs /api/clarify

[DialogFooter with close]
```

**Appearance:**
- Use existing `Dialog` / `DialogContent` / `DialogHeader` / `DialogTitle` / `DialogDescription` / `DialogFooter` components from `components/ui/dialog.tsx`
- Dialog positioned center of screen (existing `DialogContent` default: `fixed top-1/2 left-1/2`)
- Max width: `max-w-[calc(100%-2rem)]` (existing default — correct for mobile)
- Do NOT use `showCloseButton` on `DialogContent` — Phase 3 uses default `showCloseButton={true}` (X in top-right)

**DialogTitle:** "What's this item?"
- Font: existing `font-heading text-base leading-none font-medium` (inherited from `DialogTitle` component — do not override)

**DialogDescription:** AI's best expanded guess for the item — e.g. "Chicken Sandwich (Large)". If the AI returned an empty string, show "Unknown item". This is read-only in the header; the edit field appears after the user taps "Type name".
- Font: existing `text-sm text-muted-foreground` (inherited from `DialogDescription`)

**The two choice buttons:**
- Both use `variant="outline"` shadcn Button
- Both height: `h-14` (56px) — larger than the standard h-12, communicates "primary choice" within the dialog
- Both width: `w-full`
- Stack vertically with `gap-3` between them
- Positioned between the `DialogHeader` and `DialogFooter`

Button 1 — "Type name":
- Icon: `Pencil` (lucide-react), 20px, `gap-2`, left of label
- Label: "Type name"
- On tap: replaces the two-button view with the edit field view within the dialog (no dialog close/reopen — swap internal state)

Button 2 — "Take menu photo":
- Icon: `Camera` (lucide-react), 20px, `gap-2`, left of label  
- Label: "Take menu photo"
- On tap: triggers hidden `<input type="file" accept="image/*" capture="environment">` — same pattern as the bill scan input
- While `/api/clarify` is running: show a loading state within the dialog (see Surface H)

**Internal dialog states (managed by local React state — not Zustand):**

| State | What the user sees |
|-------|--------------------|
| `choices` (initial) | Two buttons: "Type name" / "Take menu photo" |
| `editing` | Edit field pre-filled with AI's best guess + Save button |
| `clarifying` | Spinner + "Checking the menu…" — camera taken, waiting for /api/clarify |
| `clarify-done` | Edit field pre-filled with clarify result + Save button |

Transition rules:
- `choices` → `editing`: user taps "Type name"
- `choices` → `clarifying`: user selects a menu photo
- `clarifying` → `clarify-done`: /api/clarify resolves (success or failure — always populate field)
- `clarifying` → `clarify-done`: /api/clarify fails — field populated with AI's best guess from expansion (D-09)
- `editing` or `clarify-done` → dialog closes: user taps Save → `updateItem()` called → badge dismissed

**Back navigation:** When in `editing` state, show a plain text link "← Back" above the edit field (`text-sm text-zinc-500`, no button variant) that returns to the `choices` state. The dialog remains open.

**Save button (in `editing` and `clarify-done` states):**
- `variant="default"` (primary fill — amber-600 via CSS variables)
- Height: `h-12`
- Width: `w-full`
- Label: "Save"
- Placed in `DialogFooter` area or directly beneath the edit field (use `DialogFooter` with `showCloseButton={false}`)
- On tap: calls `updateItem(item.id, editedName, item.priceCents)`, closes dialog, badge dismissed

**Edit field in dialog:**
- shadcn `Input` component
- `placeholder="Item name"` — same as existing inline edit
- Pre-filled with AI's best expanded name (or clarify result)
- `h-12` height (not h-10 — larger touch-friendly within dialog)
- `text-base` (16px)
- `maxLength={100}` — same as inline edit constraint

**Accessibility:**
- `DialogTitle` is the accessible name for the dialog — correct by default
- `DialogDescription` carries the AI's best guess — screen reader reads it after the title
- The two choice buttons carry self-describing labels — no extra `aria-label` needed
- Hidden camera input: `aria-hidden="true"` and `tabIndex={-1}` (same pattern as OCR input)
- In `editing` state: `autoFocus` on the Input so keyboard users can type immediately

### Surface H — Within-Dialog Loading State (Clarify)

**What:** Shown inside the disambiguation dialog while `/api/clarify` is running after the user takes a menu photo. NOT a full-screen overlay — the dialog itself shows a loading state to keep the context clear ("I took a photo and am waiting for a result").

**Appearance:**
- Dialog remains open and visible
- The two-button view is replaced by a centered column: spinner + message
- Spinner: `LoaderCircle` lucide-react, 32px (smaller than full-screen overlay — it is contained), `text-zinc-500`, `animate-spin`
- Message: "Checking the menu…" — 14px, weight 400, `text-zinc-500`, `mt-3` below spinner
- The dialog X close button remains visible so user can escape if it hangs

**Layout:**
```css
/* Replace two-button view with: */
.clarify-loading {
  display: flex;
  flex-direction: column;
  align-items: center;
  justify-content: center;
  min-height: 96px; /* same height as the two-button stack — prevents dialog resize jump */
  gap: 12px;
}
```

Tailwind equivalent: `flex flex-col items-center justify-center min-h-24 gap-3`

**No separate toast or error on clarify failure** — failure silently falls through to `clarify-done` state with the field pre-filled with the AI expansion best guess (D-09). The user sees the edit field and can correct manually. The copy below the field ("AI suggested a name — edit as needed") covers this case.

## Updated Step 2 Interaction Contract (Phase 3 additions)

This section extends the Phase 2 Step 2 interaction contract.

### New interactions added to Step 2

| Trigger | Outcome |
|---------|---------|
| OCR completes (`ocrStatus` → `'done'`) | `expandStatus` → `'loading'`; expansion overlay (Surface E) mounts immediately |
| `/api/expand` resolves | `expandStatus` → `'done'`; overlay unmounts; items updated with `displayName` + `confidence`; "Review" badges appear on low/ambiguous rows |
| `/api/expand` fails or times out | `expandStatus` → `'error'`; overlay unmounts; raw OCR names kept as-is; error toast: "Couldn't expand item names — you can edit them manually" (4s, same toast pattern as OCR error) |
| Tap item row with "Review" badge | Disambiguation dialog (Surface G) opens in `choices` state |
| Tap "Type name" in dialog | Dialog switches to `editing` state; Input auto-focuses with AI's best guess pre-filled |
| Tap "Take menu photo" in dialog | Camera input triggered; photo taken; dialog switches to `clarifying` state (Surface H) |
| `/api/clarify` resolves | Dialog switches to `clarify-done` state; Input pre-filled with clarify result (or best guess on failure) |
| Tap "Save" in dialog | `updateItem()` called; dialog closes; badge dismissed on that item |
| Tap X on disambiguation dialog | Dialog closes; badge remains (user chose not to resolve) |
| Tap "Review" item row (no dialog open) | Dialog opens fresh in `choices` state — prior `editing` local state is reset |

### Step 2 layout with expansion complete and review items present

```
[Step heading: "Items"]
[Bill thumbnail — 96px tall card]
[Items list — all editable]
  [Item row: Burger                [Review] $12.00  [Trash]]   ← low confidence
  [Item row: Chicken Sandwich (L)          $14.50   [Trash]]   ← high confidence, no badge
  [Item row: Diet Coke             [Review] $3.00   [Trash]]   ← ambiguous
[+ Add item (dashed row)]
[Assign items →]                                               ← amber-600
```

### Step 2 layout during expansion loading (immediately after OCR)

```
[Full-screen overlay: LoaderCircle + "Expanding names…"]       ← blocks all below
[Step heading: "Items"]
[Bill thumbnail — 96px tall card]
[Items list — raw OCR names, not yet expanded]
[+ Add item (dashed row)]
[Assign items →]
```

## Copywriting Contract — Phase 3 Additions

| Element | Copy | Notes |
|---------|------|-------|
| Expansion overlay message | "Expanding names…" | Present progressive. Matches "Scanning your bill…" rhythm. Ellipsis communicates ongoing process. |
| Expansion error toast | "Couldn't expand item names — you can edit them manually" | Problem + recovery path. No user blame. Consistent em-dash pattern with OCR error toast. |
| "Review" badge label | "Review" | Single word. Noun/imperative hybrid — readable as both "needs review" and "tap to review". Not "Fix" (alarming), not "?" (cryptic). |
| Disambiguation dialog title | "What's this item?" | Conversational question. Positions the AI as uncertain, not wrong. Invites user participation. |
| Disambiguation dialog description | {AI's best guess, e.g. "Chicken Sandwich (Large)"} | Dynamic. If empty: "Unknown item". No prefix like "AI thinks this is:" — just the name. |
| "Type name" button label | "Type name" | Verb + noun. Communicates the action, not the modality. |
| "Take menu photo" button label | "Take menu photo" | Verb + object. Parallel to "Scan bill" established in Phase 2. |
| Edit field hint (below input, 12px zinc-400) | "AI suggested a name — edit if needed" | Explains why the field is pre-filled. Gives user permission to change it. 12px is below the declared type scale; acceptable for hint text beneath a field (same pattern used by shadcn Input description). |
| Within-dialog loading message | "Checking the menu…" | Present progressive. Distinct from "Expanding names…" so user knows this is a different, focused action. |
| "Back" link in dialog (editing state) | "← Back" | Plain text navigation. Unicode arrow avoids icon dependency for a non-button element. |
| Save button label | "Save" | Single imperative. Unambiguous. |
| No success state copy | (none) | Badge dismissal + dialog close is the confirmation — no toast needed. The result is self-evident. |

### Destructive actions in Phase 3

No new destructive actions. Disambiguation is non-destructive — saving an edit replaces a name but does not delete data. The existing item removal Dialog from Phase 1 covers Review-badged items without modification.

## Color

Inherited from Phases 1 & 2 with one new semantic role.

| Role | Value | Phase 3 Usage |
|------|-------|---------------|
| Dominant (60%) | `#ffffff` / `#09090b` zinc-950 | Page background; expansion overlay backdrop (`zinc-950/80`) |
| Secondary (30%) | `#f4f4f5` zinc-100 / `#18181b` zinc-900 | Dialog background (`bg-popover`); expansion error toast (`bg-zinc-900`) |
| Accent (10%) | `#d97706` amber-600 | "Assign items" CTA (unchanged); "Save" button in dialog (via CSS variable `--primary`) |
| Destructive | `#dc2626` red-600 | Remove item Dialog (unchanged from Phase 1) |
| Review (new) | `bg-amber-100`, `text-amber-700`, `border-amber-300` | "Review" badge on low/ambiguous confidence item rows — ONLY |

**Review color rationale:** Orange family (amber-100/700/300) sits between the secondary zinc palette and the accent amber-600. It is warm and visible without competing with the amber-600 accent. It reads as "attention needed" without the alarm of red-600. The amber-100 background keeps it light and badge-like (not a status pill).

**Review color reserved for:** The "Review" badge only. No other element in Phase 3 uses this treatment.

**Within-dialog spinner:** `text-zinc-500` (secondary text color) — not accent, not review. It is a neutral process indicator within a confined space, not a primary action.
